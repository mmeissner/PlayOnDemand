# Pod.Grpc.Utilities

> Bidirectional bridge between the server's `IResult` / `UserError` business-error pattern and gRPC's `RpcException` / `StatusCode` / `Metadata` model.

## Purpose

The server expresses business failures as `Pod.Data.Infrastructure.IResult` — a dictionary of `UserError` enum keys to error message lists, populated by validators like `result.ArgNotNull(...)`. gRPC has no concept of structured business errors; it has `StatusCode` (~16 well-known integers) and free-form `Metadata` trailers.

This project is the only place that knows how to translate one into the other. Server services convert at the boundary (`throw result.ToException()`); client services do the inverse (`catch (RpcException ex) { return ex.ToResult(); }`). That keeps every other layer free of gRPC types **and** lets specific `UserError` enums survive the round-trip — a `UserError.StationNotFound` raised in `Pod.Services.ConnectService` lands in the kiosk's `ConnectService.cs` as the same `UserError.StationNotFound`, not as a generic `StatusCode.Unknown`.

## Responsibility

**IS** responsible for:
- `IResult.ToException()` — pick the most expressive `StatusCode`, encode every error into `Metadata` trailers as `no{n}.{userErrorName}` keys.
- `RpcException.ToResult()` — parse those trailers back into a typed `Result` with the original `UserError` keys preserved.

**IS NOT** responsible for:
- Defining `UserError` itself (that's `Pod.Enums`).
- Defining `IResult` / `Result` (that's `Pod.Data.Infrastructure`).
- Calling either method — services and clients call them at the gRPC boundary.

## Public API surface

`static class ErrorToException` (namespace `Pod.Grpc.Utilities`):

- `RpcException ToException(this IResult result)` — for use in server `*ServiceGrpc.cs` files: `if (result.HasError()) throw result.ToException();`.
- `IResult ToResult(this RpcException rpcException)` — for use in kiosk-side service wrappers: parses `rpcException.Trailers` back into `Result` entries.

## The mapping

`UserError` values are bucketed into gRPC `StatusCode` by a big switch. Highlights:

| `UserError` group | `StatusCode` |
|-------------------|--------------|
| `InternalError` | `Internal` |
| `StationInvalidQrCode`, `StationInvalidControlMode`, `OrderAlreadyPayed`, `OrderAlreadyExpired`, `OrderMaximumActiveAmountReached`, `ApplicationStationNotFound` (and similar precondition errors) | `FailedPrecondition` |
| `Station*` (`InvalidDisplayName`, `InvalidUserId`, `InvalidStationId`), `Order*` invalid-fields, `ShellServer*`, `ShellClientInvalidDeviceIdentity` | `InvalidArgument` |
| `StationNotFound` | `NotFound` |
| `ShellClientInvalidStationId`, `ShellClientInvalidPassword` | `Unauthenticated` |
| `ShellClientNoServerAvailable` | `ResourceExhausted` |
| `ShellClient*Mismatch`, `ShellClient*ConnectionTimedOut`, `ShellClientOtherConnectionStillAlive`, `ShellClientOtherDeviceIdentityConnected` | `FailedPrecondition` |
| anything else | `Unknown` |

The **last matched case wins** — if a single `IResult` carries multiple errors, only the last one in iteration order determines the `StatusCode`. Every error message is also written into the trailers as `no{n}.{lowercaseUserErrorName} = "<message>"`, so all of them survive even if only one drives the status. The decode side enumerates all trailers and reconstructs all entries.

## Notable patterns / gotchas

- **Status code is "best effort", trailers are authoritative.** A caller that just inspects `RpcException.StatusCode` will only see one error. Anything that cares about precise diagnosis must `.ToResult()` and inspect the dictionary.
- **Trailer key format is `no1.shellclientinvalidpassword`.** The leading `no{n}.` prefix exists to disambiguate multiple errors of the same kind. The decoder splits on `.` and `Enum.TryParse` is `ignoreCase: true`.
- **Unknown trailer keys** (no `.` separator) are added as `UserError.InternalError`.
- **If `ToResult` finds zero parseable trailers**, it adds the raw `RpcException.Message` as `UserError.InternalError` so the caller never gets an empty success result.
- The class is named `ErrorToException` but contains both directions — the file name `ErrorToStatus.cs` matches neither. Don't grep for `ToResult` and miss this file.
- **Newer/changed `UserError` values not in the switch** silently degrade to `StatusCode.Unknown`. When you add a new `UserError`, also add a case here or your business error will lose its semantic status code.
