# Pod.Grpc.ConnectHost.Server

> Server-side implementation of `ConnectHostServiceGrpc` — the station-discovery RPC. Single endpoint: a station calls `GetHost`, receives a `ConnectionId` plus the host/port of the Shell server it should subsequently talk to.

## Purpose

PoD supports multiple "Shell servers" (potentially clustered), and a station does not know in advance which one will own its session. The kiosk's first call after start-up is **always** `ConnectHost.GetHost(IdentityId, MaxInterfaceVersion, ReconnectConnectionId?)`. The response is a `ShellServerResponse` with the freshly minted `ConnectionId` (or the same one if reconnecting), the network endpoint of an available Shell server, and the `RequiredInterfaceVersion` that endpoint speaks.

This project is **just** the gRPC binding. All of the actual logic — picking a server, allocating a connection id, validating credentials, enforcing interface version compatibility — lives in `Pod.Services.ConnectHost.ConnectService`.

## Responsibility

**IS** responsible for:
- Implementing the `ConnectHostServiceGrpc.ConnectHostServiceGrpcBase` abstract class.
- Declaring the `[Authorize(AuthenticationSchemes = GrpcMetadataAuthenticationHandler.SchemeName)]` policy on the class so the AspNetCore pipeline runs `GrpcMetadataAuthenticationHandler` before any method body executes.
- Per-call DI scoping (each call creates its own `IServiceScope`).
- Resolving the authenticated `ClientCredentials` from the gRPC call context (now sourced from the `ClaimsPrincipal` set by the auth handler — see `CallContextUtil.ToClientCredentials`).
- Translating `ConnectService.RequestServer` results into the wire response, mapping errors to `RpcException` via `IResult.ToException()`.

**IS NOT** responsible for:
- Verifying the station password — that happens once, up-front, in `GrpcMetadataAuthenticationHandler` (which delegates to the same `Pod.Services.Extensions.VerifyCredentials` PBKDF2 path that the legacy in-method check used).
- Choosing which Shell server to return (that's `ConnectService` + `ConnectHost` orchestration in `Pod.Services`).
- Persisting the connection id (handled inside `ConnectService` via `PodDbContext`).
- Hosting the gRPC endpoint itself (that's `Pod.Web.Center/Startup.cs` via `services.AddGrpc()` + `endpoints.MapGrpcService<ConnectHostServiceGrpc>()`).
- Notifying anyone — `GetHost` does not publish on `PublisherHub<T>`. The notification stream is opened later via `ShellHostServiceGrpc.GetNotifications`.

## The single RPC: `GetHost`

```csharp
public override async Task<ShellServerResponse> GetHost(ShellServerRequest request, ServerCallContext context)
{
    using (var scope = _serviceProvider.CreateScope())
    {
        var credentials   = context.ToClientCredentials();
        var connectService = scope.ServiceProvider.GetRequiredService<ConnectService>();
        var result = await connectService.RequestServer(
            credentials,
            request.IdentityId,
            request.MaxInterfaceVersion,
            request.ReconnectConnectionId.ToGuidNullable());
        if (result.IsSuccess()) return result.ReturnValue;
        throw result.ToException();
    }
}
```

| Step | What happens |
|------|--------------|
| 0 | **Before any method-body code runs**, `GrpcMetadataAuthenticationHandler` (registered via `AddGrpcStationMetadata()` in `Startup.cs`) reads `identity`/`password` from the HTTP/2 headers, calls `IGrpcStationCredentialVerifier.VerifyAsync(...)` (PBKDF2 against `Station.PasswordHash`), and on success builds a `ClaimsPrincipal` carrying the StationId. On any failure path it returns `AuthenticateResult.Fail(...)`, which the gRPC pipeline translates into `Status(Unauthenticated, ...)` on the wire. |
| 1 | `context.ToClientCredentials()` (from `Pod.Grpc.Base.Server`) — reads the StationId from the authenticated principal (no longer re-parses headers). The Password is still pulled from headers so the returned `ClientCredentials` shape is unchanged for downstream code. |
| 2 | New DI scope so `ConnectService` and its `PodDbContext` get fresh per-call instances. |
| 3 | `ConnectService.RequestServer(credentials, identityId, maxInterfaceVersion, reconnectConnectionId?)` does the real work: enforces interface compatibility, picks/creates a `ShellServer` row, returns an `IResult<ShellServerResponse>`. (Credential verification still happens internally as defence-in-depth, but it's now a redundant re-check; auth is the source of truth.) |
| 4 | On success the inner `ShellServerResponse` is returned directly. On error, the result is thrown as a typed `RpcException` (status code derived from `UserError`; full error trailers preserved for the client to decode). |

The result-chain pattern (`credentials.VerifyCredentials<T>(_podContext)`, `result.ArgNotNull(...)`, `result.ValueTrue(...)`) lives inside `ConnectService.RequestServer`, not here.

## Notable patterns / gotchas

- **Per-call DI scope.** Under Grpc.AspNetCore the service class itself is registered scoped automatically by `MapGrpcService<T>()`. The explicit `using (_serviceProvider.CreateScope())` inside the method is still needed for `ConnectService` + `PodDbContext` because they're resolved from the singleton root provider; without the scope you'd get cross-call DbContext bleed. (When the orchestrator simplifies the IoC layout this can be tightened.)
- **`[Authorize]` is on the class, not the method.** The single attribute covers every RPC the service exposes, current and future.
- **No `PublisherHub<T>` publish.** Unlike `ShellHostServiceGrpc`, this service is purely request/response — no streams, no notification side-effects.
- **`ReconnectConnectionId` is optional** on the wire (`GuidAsBytes`), and the `ToGuidNullable()` extension correctly returns `null` if the bytes are empty. `ConnectService` uses that null/non-null distinction to decide between "first connect" and "reconnect existing connection".
- **`MaxInterfaceVersion` enforces compatibility**: if the kiosk advertises a version older than the server's `RequiredInterfaceVersion`, `ConnectService.RequestServer` returns an error. `RequiredInterfaceVersion` is round-tripped in the response so the kiosk can decide whether to upgrade.
- **No streaming RPCs here.** All notification work is on `ShellHostServiceGrpc.GetNotifications`.
