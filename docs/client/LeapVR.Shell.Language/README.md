# LeapVR.Shell.Language

> Kiosk localisation tier: the runtime `LanguageSelector` that flips `CultureInfo.CurrentUICulture` and `WPFLocalizeExtension`'s active culture, plus the bundled `Resources.resx` translations and XLIFF translation units for English and Simplified Chinese.

## Purpose

Kiosks are deployed in mixed-language regions; this project owns the runtime mechanics of switching between supported languages and the resource files that hold the actual strings.

`LanguageSelector`:

1. Reads `SystemConfig.DefaultLanguage` and `SupportedLanguageCultureNames` via `IConfigFileRepository<SystemConfig>` on construction.
2. Sets `CurrentCulture` and `CurrentUICulture` for the process.
3. Sets `WPFLocalizeExtension.Engine.LocalizeDictionary.Instance.Culture` so all XAML `LocText` bindings re-evaluate.
4. Publishes a `UILanguageChangedEvent` on `IUIMessageBroker` whenever the culture changes, so view models can refresh derived display strings.
5. Persists changes back to `SystemConfig` when `SetDefaultCulture(culture)` is called.

## Responsibility

**It IS responsible for:**
- Implementing `ILanguageSelector` (interface in `LeapVR.Shell.Domain.Models.Language`).
- Persisting and restoring the active language.
- Owning the project-wide `Resources.*.resx` strings used everywhere except `LeapVR.Shell.Categories` (which has its own `Resource.*.resx`).
- Publishing `UILanguageChangedEvent` so the UI can re-render.

**It is NOT responsible for:**
- Localising category labels (those live in `LeapVR.Shell.Categories`'s own resource files).
- Translation tooling itself — the XLIFF files in `MultilingualResources/` are generated by Visual Studio's Multilingual App Toolkit.

## Public API surface

| Type | Purpose |
|---|---|
| `LanguageSelector` | Implements `ILanguageSelector`. Properties: `DefaultCulture`, `SupportedCultures`, `CurrentCulture`. Methods: `ActivateDefaultCultureInfo`, `ActivateCultureInfo(string)` / `(CultureInfo)`, `SetDefaultCulture(CultureInfo)`. |
| `Resources` (auto-generated `Resources.Designer.cs`) | Strongly-typed accessor for the `.resx` strings. |

Resource files:

- `Resources.resx` — invariant-culture default (typically English text used at fallback).
- `Resources.en-US.resx` — English overrides.
- `Resources.zh-CN.resx` — Simplified Chinese overrides.
- `MultilingualResources/Resources.en-US.xlf` (and `.zh-CN.xlf`) — XLIFF translation units used by tooling.

## Notable patterns / gotchas

- **`SupportedCultures` is set once at construction** from `SystemConfig.SupportedLanguageCultureNames`. To change supported languages at runtime you'd need to rebuild the `LanguageSelector`. Today it's a singleton — restart-only change.
- **Persisting a new default writes to `SystemConfig`** via `IConfigFileRepository<SystemConfig>` — the same JSON-on-disk file the wizard wrote during setup.
- **WPF binding refresh is via `WPFLocalizeExtension.Engine.LocalizeDictionary.Instance.Culture`.** Setting this re-evaluates every `LocText`/`LocVisibility`/etc. binding in the visual tree. View models that derive *manual* strings should subscribe to `UILanguageChangedEvent` and re-compute.
- **Two `.resx` projects coexist** — this one (`LeapVR.Shell.Language`) for general UI text, and `LeapVR.Shell.Categories` for category labels. They use the same `WPFLocalizeExtension` engine but have separate resource scopes addressed by assembly + key.
- **The setter for the default language silently catches and logs exceptions** (`SetDefaultCulture` returns `false` on failure). Don't rely on exceptions to detect a bad culture name; check the return value.

## Consumers

- `LeapVR.Shell` — `Bootstrapper.RegisterHostSystem` registers `ILanguageSelector` as singleton; `LanguageSelectViewModel` (admin → settings) drives changes.
- `LeapVR.Shell.Setup` — also registers `ILanguageSelector` so the wizard UI is localised.
- Pretty much every XAML view in `LeapVR.Shell/UI/...` indirectly via `LocText` markup extensions.
